# 第7章：多模态与工具化——视觉语言模型、Function Calling、Code Generation

> **本章动机（养老机器人视角）**  
> 一个陪护机器人要完成"帮老人找到红色药盒并提醒服药"这个任务，它需要同时"看"（摄像头识别红色药盒的位置）、"听"（语音转文字理解指令）、"做"（调用家电控制接口打开灯、调用医疗记录系统查询服药时间）。  
> VLM 是它的"眼睛"，Function Calling 是它调用外部系统的"手"，Code Generation 是它临时编写数据分析脚本的"脑内自言自语"。  
> 但一个深层问题是：**为什么视觉和语言需要被"融合"到一个模型中？为什么不能让视觉模型和语言模型各做各的，通过文本中转？** 这个问题没有直觉上的简单答案——2020年的研究者确实尝试过"视觉模型输出文本描述，语言模型处理文本"的串行方案，但它失败了。为什么失败？失败的原因是原理性的还是工程性的？本章将带你还原这个决策过程。

---

## 核心思想

大语言模型的能力边界不仅取决于参数规模，更取决于它能接入的感知模态与外部工具。多模态扩展（将视觉、音频映射到语言空间）为 LLM 提供了"眼睛"，工具化（Function Calling、API 调用）为 LLM 提供了"手"——这两者共同构成了 LLM 从"对话者"变为"行动者"的关键一跃。

但"关键一跃"这个表述需要被严格审视。**多模态融合是否真的让模型"理解"了视觉内容，还是只是在做更复杂的模式匹配？** [开放问题]。**工具调用是否真的让模型"行动"了，还是只是生成了结构化文本让外部系统去执行？** [合理推断]——后者更接近事实。理解这些区分，是构建生产级 Agent 系统的认知前提。

本章将围绕三个核心问题展开：
1. 多模态融合的三大架构（投影层、交叉注意力、统一编码器）各自在什么约束下被选择？它们的 trade-off 是什么？
2. 工具调用（Function Calling）的工程化协议是如何从自然语言推理中"分裂"出来的？
3. Code Generation 作为"通用工具"，与 Function Calling 相比有哪些独特的优势和风险？

## 历史脉络：从独立模态到统一行动接口——不是"自然融合"，而是"各自撞墙后的被迫转向"

### 2019-2020：双塔架构的辉煌与边界

2019-2020 年，多模态 AI 的主流范式是"双塔架构"：独立的视觉编码器和文本编码器通过对比学习对齐（如 CLIP）。CLIP 的核心思想是——让图像和描述它的文本在共享的向量空间中靠近，不相关的图像-文本对远离。这种架构在图像检索和零样本分类上表现优异，但研究者很快发现了一个**原理性局限**：双塔架构无法处理需要**深层视觉-语言推理**的任务。

例如，回答"图片中有几个人，他们在做什么？"需要理解图像的语义内容（有什么物体）、空间关系（物体之间如何关联）和动作识别（人在做什么）。CLIP 的对比学习只建立了**图像级别的语义对应**（这张图和这段文字相关），而非**区域级别的语义推理**（图中的左上角有一个人在看书）。这不是工程上的不足，而是架构上的结构性限制——双塔的信息交汇点仅在最后的对比损失层，缺乏中层交互。

**[已证明]**：CLIP 在零样本 ImageNet 分类上达到了 76.2% 的 top-1 准确率 [Radford et al., 2021]，但在需要细粒度视觉推理的 VQA（视觉问答）任务上，纯 CLIP 架构的表现显著低于融合架构。这是架构选择决定能力上限的经典案例。

### 2021-2022：交叉注意力的尝试与代价

2021-2022 年，BLIP 和 Flamingo 展示了另一种路径：将视觉编码器与语言模型通过**交叉注意力**连接，让语言模型在生成文本时直接"查看"图像特征。这种架构允许模型进行复杂的视觉推理——Flamingo 能回答关于图像的多轮问题，BLIP-2 能进行图像描述和视觉问答。

但交叉注意力的训练成本高昂：需要在大规模图像-文本对上进行端到端训练，且交叉注意力模块需要从头学习视觉-语言的交互模式。在2021年，训练一个 Flamingo-80B 需要数百万美元的 GPU 资源——这远非学术研究者可以承受的。

### 2023：投影层架构的"民主化"

2023 年，LLaVA 和 MiniGPT-4 提出了更轻量的方案：使用预训练的视觉编码器（如 CLIP 的 ViT）提取图像特征，然后通过一个简单的**线性投影层**将视觉特征映射到语言模型的输入空间。这种"冻结视觉编码器 + 轻量投影 + 微调语言模型"的策略大幅降低了训练成本——训练一个 LLaVA-7B 只需要 8 张 A100 训练约 2 天，成本不到 500 美元。这使得学术研究者也能训练视觉语言模型（VLM）。

**关键问题**：为什么投影层架构——一个数学上如此简单的方案（一个线性变换矩阵 W ∈ R^{d_l × d_v}）——能在很多任务上达到接近交叉注意力架构的效果？这是因为**预训练的视觉编码器已经提取了高质量的视觉特征**，而语言模型本身具有强大的推理能力——投影层只需要将视觉特征"翻译"到语言模型能理解的空间。但这也有代价：投影层架构在需要精细视觉理解的任务（如读取图像中的小字、理解空间关系）上表现较差。

### 工具化的平行发展

与此同时，工具化也在平行发展。2022 年的 Toolformer [Schick et al., 2023] 首次证明 LLM 可以在预训练阶段学习调用 API（如搜索引擎、计算器、翻译器）。2023 年，GPT-4 的 Function Calling 能力将工具使用标准化为 JSON 格式的结构化输出——这是一个工程化里程碑，而非理论突破。ToolLLM [Qin et al., 2024] 进一步将工具使用扩展到 16000+ 真实 API，并提出了系统化的工具学习框架。Code Generation 则是一个特殊的"工具化"形式：LLM 生成可执行代码，从而间接操控计算环境。

这三条线的交汇点在于：一个能"看见"世界（多模态）、能"使用工具"（工具化）、能"编写代码"（Code Generation）的 LLM，已经具备了 Agent 的基本能力。第8-11章将讨论如何将这些能力组织成系统化的 Agent 架构。

**但请注意**：这三条线的"交汇"不是研究者的精心设计，而是各自独立发展后的"意外相遇"。多模态研究者最初的目标是视觉理解，工具化研究者的目标是扩展 LLM 的能力边界，Code Generation 研究者的目标是自动化编程。它们的交汇是因为 Agent 的需求把它们拉到了一起——这是一个**需求驱动的后验整合**，而非**设计驱动的前验规划**。[作者推测]

## 7.1 多模态融合的三大架构——在约束空间中的理性选择

### 问题本质

多模态融合要解决的根本问题不是"如何把图像和文本拼在一起"，而是：**一个以文本为母语的模型（LLM），如何接收并理解非文本的感知信号？** 这本质上是一个**跨模态翻译问题**——将视觉模态的信息翻译为语言模型可以处理的表征。

### 历史约束：2020-2023年的世界是什么样的？

- **计算约束**：2020年，训练一个 Flamingo-80B 需要大量的 TPU 资源，只有 Google/DeepMind 级别的机构能承担。2023年，LLaVA 的出现将成本降低到学术可承受范围，但训练一个 70B+ 的 VLM 仍然需要多卡 A100。
- **数据约束**：高质量的图像-文本对（如 LAION-5B）在2021年后才大规模可用。在此之前，CC3M 等数据集的规模限制了 VLM 的训练。更重要的是，**视觉指令数据**（如"看图回答问题"格式的数据）严重匮乏——LLaVA-Instruct-150K 是首批大规模视觉指令数据集之一。
- **理论约束**：2020年之前，"如何将视觉特征映射到语言空间"这个问题没有标准答案。CLIP 提供了对齐后的视觉表征，但 CLIP 的输出空间和 LLM 的输入空间并不天然兼容——它们的统计分布不同、维度不同、语义粒度不同。
- **社区约束**：2022年之前，视觉-语言社区和 NLP 社区相对独立。VLM 的先驱工作（如 ViLBERT, LXMERT）来自视觉社区，它们的架构设计和训练理念与 LLM 社区有显著差异。LLaVA 的贡献之一恰恰是**用 NLP 的范式（指令微调）来做视觉任务**——这种跨社区的方法迁移本身就是一种创新。

### 方案空间：在当时约束下，有哪些候选路径？

#### 路径A：双塔架构 + 文本中转（CLIP + LLM 串行）

- **核心思想**：视觉编码器（CLIP）先将图像编码为向量，然后通过一个文本描述生成器将向量"翻译"为文本描述，最后将文本描述输入 LLM 处理
- **在当时约束下的优势**：架构极其简单，不需要修改 LLM 的内部结构；两个模块可以独立训练和升级
- **在当时约束下的劣势**：**信息瓶颈**——文本描述是一个极度压缩的通道。一张 1024×1024 的图像包含数百万个像素的信息，但一段 50 词的图像描述只能传达其中的极小部分。例如，"一只猫坐在垫子上"这句话丢失了猫的品种、颜色、姿态、垫子的位置、背景环境等大量信息
- **反事实推演**：如果这条路径被广泛采纳，短期（1-2年）内它能在简单视觉问答上工作（如"图中有猫吗？"），但长期（3-5年）在需要精细视觉理解的任务（如"图像左上角的文字是什么？"、"红色杯子在蓝色杯子的左边还是右边？"）上会遭遇原理性瓶颈。事实上，2022年的部分系统采用了这种方案，用户很快发现 LLM 的回答"看起来合理但视觉细节错误"——因为 LLM 处理的是文本描述，而非图像本身
- **认知偏差警示**：这条路径的拥护者可能陷入了**工具偏差**——因为手上有强大的 LLM（文本处理能力极强），所以倾向于把所有输入都转化为文本。但视觉信息的丰富度远超文本的表达能力——这是一个信息论事实，不是工程问题

#### 路径B：交叉注意力架构（Flamingo, BLIP-2）

- **核心思想**：在语言模型的 Transformer 层中插入交叉注意力模块，让语言模型的每一层都可以直接 attending 到视觉特征
- **在当时约束下的优势**：**深层交互**——视觉信息在语言模型的每一层都参与计算，可以捕捉复杂的视觉-语言关系；灵活性强，可以处理任意数量的视觉 token（如多图像输入）
- **在当时约束下的劣势**：训练成本高——需要端到端训练视觉编码器和语言模型，或至少训练交叉注意力模块；推理延迟——每层额外的交叉注意力增加了计算量
- **反事实推演**：如果交叉注意力架构成为当时的主流选择，学术研究者将被排除在 VLM 研究之外（因为训练成本不可承受），VLM 的发展可能更加集中在大公司。但交叉注意力在复杂视觉推理上的优势意味着它在特定领域（如医学图像分析、文档理解）可能长期保持最优——事实上2025年这仍然是这样的
- **认知偏差警示**：交叉注意力的拥护者可能陷入**复杂度偏差**——认为"更深的交互一定更好"。但2023年的实验表明，在很多视觉问答任务上，投影层架构（LLaVA）的性能接近交叉注意力架构（BLIP-2），而训练成本低一个数量级。这意味着交叉注意力的"深层交互"在许多任务上并非必需——简单的特征映射 + LLM 的推理能力已经足够

#### 路径C：投影层架构（LLaVA, MiniGPT-4）

- **核心思想**：使用预训练的视觉编码器提取图像特征，通过一个简单的线性投影层将视觉特征映射到语言模型的输入空间，视觉 token 和文本 token 拼接后一起输入 LLM
- **在当时约束下的优势**：训练成本低——视觉编码器冻结，只需训练投影层和语言模型的 adapter；架构简单——不需要修改语言模型的内部结构
- **在当时约束下的劣势**：能力受限——视觉信息只通过线性投影进入语言模型，复杂的视觉-语言交互（如"图像左上角的文字是什么？"）可能处理不好；分辨率限制——CLIP ViT 通常使用固定分辨率（如 224×224），对于高分辨率图像（如 4K 照片），细节信息丢失严重
- **反事实推演**：如果投影层架构没有被提出，VLM 的民主化可能会推迟 1-2 年。大公司的交叉注意力模型（如 Gemini）仍然会发展，但学术界的参与度会大幅降低，创新速度可能放缓。LLaVA 的贡献不仅是技术上的，更是社会学上的——它证明了"简单方案 + 开源生态"可以与"复杂方案 + 闭源资源"竞争
- **认知偏差警示**：投影层的拥护者可能陷入**简化偏差**——认为"简单方案在所有场景下都够用"。但2024年的研究表明，在需要精细视觉理解的任务（如 OCR、图表理解、空间关系推理）上，投影层架构的性能显著低于交叉注意力和统一编码器架构。选择架构时必须考虑任务需求

#### 路径D：统一编码器架构（GPT-4V, Gemini）

- **核心思想**：将文本和图像统一处理为 token 序列，用同一个 Transformer 编码。图像被分割为 patch（如 16×16 像素），每个 patch 线性投影为向量；文本被 tokenize 为嵌入向量；视觉 token 和文本 token 拼接为一个序列
- **在当时约束下的优势**：架构简洁——不需要额外的交叉注意力或投影层，文本和视觉完全对称处理；天然支持多图像——多个图像的 patch 可以像文本 token 一样拼接在序列中
- **在当时约束下的劣势**：数据需求极大——统一编码器需要从头学习视觉-语言的交互关系，需要比投影层架构更多的训练数据；计算开销大——视觉 patch 的数量通常很大（如 1024×1024 图像有 4096 个 16×16 patch），导致输入序列很长，注意力计算昂贵
- **反事实推演**：如果统一编码器在2022年就被证明最优，那么只有拥有超大规模数据和算力的机构（OpenAI, Google）能参与 VLM 竞争。开源社区可能需要等到 2025 年后（当开源基础模型足够强大时）才能跟进。这会使得 VLM 领域的创新高度集中，多样性降低
- **认知偏差警示**：统一编码器的拥护者可能陷入**统一性偏好**——认为"一个模型处理所有模态"在原理上优于"多个模块组合"。但这种偏好可能是一种美学偏好而非实证结论。2025年的证据[强实验证据]表明，在许多任务上，投影层架构（组合方案）的性能可以接近统一编码器——"统一"并不天然意味着"更好"

#### 选择逻辑

在2023年的约束下，投影层架构（LLaVA）之所以成为学术主流，不是因为它在性能上最优，而是因为它在**性能-成本 trade-off** 上最优。交叉注意力在精细视觉推理上更优，统一编码器在极限性能上更优，但两者的训练成本都远超学术可承受范围。**在资源受限的约束下，投影层架构在"可行性"维度上最优，但牺牲了精细视觉理解能力。**

工程上的共识是：对于大多数应用，投影层架构是性价比最高的选择。只有在需要处理复杂视觉推理（如文档理解、医学图像分析）时，才需要交叉注意力或统一编码器架构。但请注意——这个"共识"可能带有**可用性偏差**（availability bias）：因为投影层架构最容易被尝试和验证，所以它的"成功案例"最多，但这不代表它在所有场景下都是最优的。

### 7.1.4 架构选择的工程指南

| 架构 | 适用场景 | 训练成本 | 推理成本 | 数据需求 | 典型模型 |
|------|----------|----------|----------|----------|----------|
| 投影层 | 快速原型、资源有限、一般视觉问答 | 低 | 低 | 中等 | LLaVA, MiniGPT-4 |
| 交叉注意力 | 复杂视觉推理、多图像推理、文档理解 | 高 | 高 | 高 | Flamingo, BLIP-2 |
| 统一编码器 | 追求极致性能、大规模部署、原生多模态 | 极高 | 高 | 极高 | GPT-4V, Gemini |

**[时间敏感信息]** 上述成本评估基于2025年中的技术状态。随着更高效的训练方法（如 LoRA、QLoRA）和更强大的开源基础模型的出现，各架构的训练成本都在快速下降。

## 7.2 视觉语言模型：从 CLIP 到 LLaVA——一个"翻译"视角

### 7.2.1 为什么不能"串行"：文本中转的失败

在深入 LLaVA 的架构之前，让我们先理解一个关键的**失败案例**——为什么"视觉编码器 → 文本描述 → LLM 处理"的串行方案失败了？

#### 问题动机

2020-2021年，一些研究者和工程师尝试了这种直觉上"最简单"的方案：用图像描述模型（如 BLIP）生成图像的文本描述，然后将描述输入 LLM。这在工程上极其简单——不需要修改 LLM 的任何内部结构，只需要在前面加一个"图像描述器"。

#### 失败的原因：信息论分析

一张 224×224 的 RGB 图像包含 224×224×3 = 150,528 个像素值。即使经过压缩，视觉编码器的输出（如 CLIP ViT-B/16 的 196 个 token × 768 维 = 150,528 维）仍然保留了大量的空间和语义信息。

相比之下，一段 50 词的图像描述（约 70 个 token × 768 维 ≈ 53,760 维）在维度上可能相近，但在**信息内容**上完全不同。文本描述是一个**高度语义化的、有损的压缩**——它只保留了人类语言能表达的信息，而丢失了：

1. **空间精度**："猫在垫子上"没有说猫在垫子的哪个位置、朝向哪个方向
2. **数量信息**："图中有几个人"——描述可能只说"有人群"，而非"有5个人"
3. **属性细节**：颜色、大小、纹理等细粒度属性在文本描述中经常被省略
4. **关系结构**：物体之间的空间关系（如"A在B的左边"、"C在D和E之间"）很难用简短文本完整描述

**[已证明]**：信息论告诉我们，通过一个低带宽通道（文本描述）传输高带宽信号（图像），不可避免地存在信息损失。这不是工程问题，而是**信道容量定理**的直接推论——文本描述的信道容量不足以传输图像的全部信息。

#### 认知偏差警示：文本中心主义偏差

"文本中转"方案的拥护者可能陷入了**文本中心主义偏差**——因为 LLM 在文本上表现极好，所以倾向于把所有输入都转化为文本。但人类的认知不是这样的：我们看到图像时，并不是先在脑中生成一段文字描述，再处理这段文字——视觉信息直接参与认知过程。强行将视觉信息"降维"为文本，是一种人为引入的信息瓶颈。

#### 正确的思路：让视觉特征直接进入 LLM

LLaVA 的核心洞见是：**不要翻译成文本，而是翻译成 LLM 能理解的向量**。视觉编码器的输出本身就是高维向量——只需要一个线性变换，将视觉向量映射到 LLM 的词嵌入空间，LLM 就能"看到"图像。这就像给一个只懂英语的人配了一副"翻译眼镜"——图像信息不需要先变成英语，而是直接以他能理解的格式输入大脑。

### 7.2.2 LLaVA 的训练流程

LLaVA（Large Language and Vision Assistant）的训练分为两个阶段，每个阶段解决一个不同的问题：

**阶段1：预训练投影层——建立"翻译词典"**

使用 CC3M 等图像-文本对齐数据集，训练视觉特征到语言模型输入空间的投影层。语言模型和视觉编码器都冻结，只训练投影层。

这一步的目标是什么？**让视觉 token 和文本 token 在语言模型的输入空间中对齐**。具体来说，一张猫的图像经过视觉编码器和投影层后得到的向量，应该与"猫"这个词的词嵌入在 LLM 的输入空间中"靠近"——这样 LLM 在处理视觉 token 时，就像在处理一个"视觉词汇"。

**阶段2：视觉指令微调——教 LLM "用视觉词汇推理"**

使用视觉指令数据集（如 LLaVA-Instruct-150K），在对话格式中微调语言模型（包括投影层）。这些指令数据包含图像-问题-回答三元组，例如：

```
[图像：一只猫在垫子上]
Human: 图中有什么动物？
Assistant: 图中有一只猫，它正坐在垫子上。
```

这一步的目标不是"教 LLM 认识猫"——LLM 在预训练阶段已经知道了"猫"的概念。这一步的目标是**教 LLM 将视觉 token 与已有的文本知识关联起来**——学会"看到"视觉 token 中的猫，并调用关于猫的文本知识来回答问题。

**[合理推断]**：阶段2的微调本质上是在 LLM 的表征空间中建立**跨模态关联**——视觉表征和文本表征在预训练阶段各自形成了内部结构，微调的任务是在这两种结构之间建立"桥梁"。这个桥梁的质量取决于微调数据的多样性和质量。

### 7.2.3 视觉 token 的位置编码问题

当视觉 patch 和文本 token 拼接时，一个关键的工程问题出现：**如何给视觉 patch 分配位置编码？**

这个问题为什么重要？因为位置编码是 Transformer 理解序列结构的核心机制。文本 token 的位置编码编码的是"第1个词、第2个词..."的序列顺序，但视觉 patch 不是线性序列——它们是一张二维图像的局部块，具有**空间结构**。

#### 方案空间

**方案1：共享位置编码**
- 核心思想：视觉 patch 和文本 token 使用同一套位置编码
- 问题：忽略了视觉的空间结构——图像中相邻的 patch 在空间上相关，但位置编码只编码了序列顺序。位置1和位置2在序列中相邻，但可能对应图像中不相邻的区域
- 评估：[已证明]——实验表明这种方案在需要空间理解的任务上性能下降

**方案2：2D 位置编码**
- 核心思想：为视觉 patch 分配二维位置编码（x, y 坐标），保留空间结构。例如，对于 2×2 的图像 patch，位置编码为 (0,0), (0,1), (1,0), (1,1)
- 优势：保留了图像的二维空间结构，模型可以推断 patch 之间的空间关系
- 劣势：需要额外的位置编码维度；与文本的 1D 位置编码不兼容
- 评估：[强实验证据]——大多数成功的 VLM 采用某种形式的 2D 位置编码

**方案3：RoPE 的二维扩展**
- 核心思想：使用 RoPE（Rotary Position Embedding）的二维扩展，让模型学习视觉 patch 的相对空间关系
- 优势：相对位置比绝对位置更符合视觉感知（人类更关注"A在B旁边"而非"A在第3行第4列"）；与使用 RoPE 的 LLM 天然兼容
- 劣势：二维 RoPE 的数学形式更复杂；在极端分辨率变化时可能不稳定
- 评估：[合理推断]——理论上更优雅，但实验证据尚不充分

当前主流方案是方案2（2D 位置编码）或方案3（RoPE 扩展），因为视觉的空间结构对理解至关重要。**但请注意**：位置编码的选择目前在很大程度上是一个**工程经验**问题，而非有理论指导的设计——不同 VLM 采用不同方案，缺乏系统性的对比实验。这是一个**[开放问题]**。

### 7.2.4 分辨率与视觉 token 数量的 trade-off

一个实际工程问题：图像分辨率越高，视觉细节越丰富，但视觉 token 数量也越多，计算成本越高。

| 分辨率 | Patch 大小 | 视觉 token 数 | 适用场景 |
|--------|-----------|--------------|----------|
| 224×224 | 16×16 | 196 | 简单分类、粗略描述 |
| 336×336 | 14×14 | 576 | 文档理解、一般视觉问答 |
| 448×448 | 14×14 | 1024 | 精细视觉推理、医学图像 |
| 1024×1024 | 16×16 | 4096 | 高分辨率场景、OCR |

**关键 trade-off**：视觉 token 数量增加 → LLM 的序列长度增加 → 注意力计算 O(n²) 增长 → 显存和延迟急剧上升。

LLaVA-1.5 的 AnyRes 方案提供了一种折中：将高分辨率图像切分为多个子图（如 2×2 网格），每个子图独立编码，然后将所有子图的视觉 token 拼接。这样既保留了高分辨率信息，又控制了单次编码的计算量。但这引入了新的问题：**子图之间的空间关系如何保持？** 这是一个活跃的研究方向 [开放问题]。

## 7.3 工具使用：Function Calling 的工程化——从自然语言推理中"分裂"出的结构化接口

### 问题本质

Function Calling 要解决的根本问题不是"如何让 LLM 调用 API"，而是：**如何让 LLM 的自然语言输出与外部系统的结构化接口对接？** LLM 的输出是自然语言文本——灵活、有歧义、格式不固定。外部系统的接口是结构化的——严格的参数类型、固定的字段名、明确的错误码。Function Calling 就是这两者之间的"协议适配器"。

### 历史约束：2022-2023年的世界是什么样的？

- **LLM 能力约束**：2022年的 GPT-3.5 已经能生成"看起来像调用工具"的文本（如"我应该搜索一下..."），但输出格式不稳定——有时输出 JSON，有时输出自然语言，有时混在一起。这种不稳定性使得构建可靠的工具调用系统极其困难。
- **工程约束**：2022年的 Agent 系统（如早期的 LangChain）需要大量的 prompt 工程和输出解析代码来处理 LLM 的非结构化输出。每个工具的调用逻辑都需要单独编写解析器——这不可扩展。
- **社区需求**：随着 Agent 概念的火热，开发者迫切需要一种**标准化的工具调用协议**，使得任何 LLM 都能以统一的方式调用任何工具。

### 方案空间：在当时约束下，有哪些候选路径？

#### 路径A：Prompt 工程 + 正则解析

- **核心思想**：通过精心设计的 prompt 引导 LLM 输出特定格式的文本（如 JSON），然后用正则表达式或 JSON 解析器提取工具调用信息
- **在当时约束下的优势**：不需要 LLM 的原生支持，任何 LLM 都能用
- **在当时约束下的劣势**：**不可靠**——LLM 可能不遵循格式、输出语法错误的 JSON、在 JSON 中混入自然语言。解析失败率在实际系统中可达 5-15%
- **反事实推演**：如果这条路径继续发展，我们可能会拥有极其复杂的 prompt 模板和容错解析器。但可靠性永远无法达到生产级——因为 LLM 的输出本质上是概率性的，无法保证 100% 遵循格式
- **认知偏差警示**：这条路径的拥护者可能陷入**控制错觉**——认为"只要 prompt 写得够好，输出就会可靠"。但 LLM 的输出是概率分布的采样，不存在 100% 的格式保证

#### 路径B：微调 LLM 输出结构化格式

- **核心思想**：在工具调用数据集上微调 LLM，使其学会输出严格格式的 JSON
- **在当时约束下的优势**：比 prompt 工程更可靠；可以针对特定工具集优化
- **在当时约束下的劣势**：需要大量标注数据；微调后的模型可能**过拟合**到训练时的工具集，对新工具的泛化能力差；每次新增工具都需要重新微调
- **反事实推演**：如果微调路径成为主流，工具调用将高度定制化——每个公司都有自己的工具调用模型，互不兼容。开源生态难以形成，因为微调数据和方法都是闭源的

#### 路径C：原生 Function Calling（GPT-4, 2023）

- **核心思想**：在 LLM 的训练过程中就引入工具调用的概念，让模型原生支持输出结构化的工具调用 JSON。OpenAI 在 GPT-4 中引入了 `function_call` 接口，后续演进为 `tool_calls`
- **在当时约束下的优势**：**标准化**——统一的 JSON Schema 描述工具，统一的输出格式；**可靠性高**——因为是在训练阶段就学会了结构化输出，格式正确率远超 prompt 工程
- **在当时约束下的劣势**：需要 LLM 提供商的原生支持；工具描述占用上下文 token；模型可能"幻觉"出不存在的工具或参数
- **选择逻辑**：**在2023年的约束下，GPT-4 的原生 Function Calling 在"可靠性"和"标准化"维度上最优，但牺牲了"灵活性"（只能调用预定义的工具集）和"成本"（工具描述占用上下文）。**

#### 认知偏差警示：结构化输出的"可靠性幻觉"

Function Calling 的出现让很多开发者认为"工具调用问题已经解决了"。但这种感觉可能是一种**可用性偏差**——因为在常见场景下 Function Calling 确实可靠（格式正确率 > 95%），所以人们倾向于忽略那 5% 的失败 case。在生产系统中，5% 的失败率意味着每 20 次调用就有 1 次出错——如果是关键操作（如转账、删除文件），这是不可接受的。**Function Calling 提高了工具调用的下限，但没有消除错误。** [强实验证据]

### 7.3.1 Function Calling 的协议设计

Function Calling 的标准协议包含三个组件：

**1. 工具描述**：每个可用工具用 JSON Schema 描述，包含工具名称、参数类型和描述：

```json
{
  "name": "calculator",
  "description": "Perform arithmetic calculations",
  "parameters": {
    "type": "object",
    "properties": {
      "expression": {
        "type": "string",
        "description": "Math expression to evaluate"
      }
    },
    "required": ["expression"]
  }
}
```

**[注意]** 工具描述的质量直接影响 LLM 的调用准确率。description 字段不仅是文档——它是 LLM 用来判断"什么时候用这个工具"的关键信息。一个写得不好的 description（如"计算器"）可能导致 LLM 在不需要时调用它，或在需要时不知道调用它。

**2. 调用生成**：LLM 在生成文本时，如果判断需要使用工具，则输出一个 JSON 对象：

```json
{"name": "calculator", "arguments": {"expression": "37 + 15"}}
```

**3. 结果注入**：执行工具后，将结果注入对话上下文：

```
User: What is 37 + 15?
Assistant: {"name": "calculator", "arguments": {"expression": "37 + 15"}}
System: Tool result: 52
Assistant: The answer is 52.
```

### 7.3.2 工具调用的工程挑战

#### 参数填充：LLM 的"实体识别"挑战

LLM 需要准确地将用户请求映射到工具参数。例如，用户说"查一下北京今天的天气"，模型需要提取出 `location="北京"` 和 `date="今天"`。这涉及：

- **实体识别**：从自然语言中提取结构化信息（"北京"是地名，"今天"是日期）
- **指代消解**：如果用户先说"北京"再说"那里呢？"，模型需要知道"那里"指"北京"
- **日期解析**："今天"需要被解析为具体日期（如 2025-06-28），"下周一"需要计算相对日期
- **类型转换**：用户说"三个人"，但参数要求 int 类型，需要将"三"转换为 3

这些子任务中的任何一个出错，都会导致参数填充错误。而 LLM 在这些子任务上的表现并不均匀——实体识别通常较好（因为有大量训练数据），但日期解析和类型转换容易出错（因为需要精确的符号操作）。

#### 错误驱动学习：参数填充的常见错误

**常见错误1：类型不匹配**

```json
// 用户说"把温度调到二十度"
// 错误输出：
{"name": "set_temperature", "arguments": {"temperature": "二十"}}
// 正确输出：
{"name": "set_temperature", "arguments": {"temperature": 20}}
```

**错误分析**：LLM 是文本生成模型，它的默认输出是字符串。当参数 schema 要求 int 类型时，LLM 可能输出字符串"二十"而非数字 20。这不是 LLM"不理解"——而是它的输出空间是 token 序列，需要后处理来确保类型正确。**正确做法**：在 JSON Schema 中明确标注类型，并在解析后做类型验证和转换。

**常见错误2：幻觉参数**

```json
// 工具定义中没有"unit"参数
// 错误输出：
{"name": "set_temperature", "arguments": {"temperature": 20, "unit": "celsius"}}
```

**错误分析**：LLM 可能基于常识添加 schema 中不存在的参数。这源于 LLM 的训练数据中包含大量带有"unit"字段的温度设置场景。**正确做法**：在解析后做 schema 验证，拒绝未知参数；或在 prompt 中明确说明"只使用定义中存在的参数"。

**认知偏差根源**：这两个错误都源于一种**过度泛化偏差**——LLM 在训练数据中见过太多类似的场景，倾向于根据"通常模式"而非"当前 schema"来填充参数。这是统计学习模型的固有特性，不是 bug，而是特征——但在工具调用场景下，它变成了 bug。

#### 错误恢复：当工具调用失败时

工具调用可能失败（如 API 超时、参数错误、服务不可用）。Agent 需要能够：

1. **检测错误**：解析 HTTP 状态码或异常信息。这看似简单，但实践中 LLM 经常无法正确理解错误信息——因为错误信息的格式因 API 而异
2. **重试**：使用指数退避策略（间隔 1s, 2s, 4s, 8s），最多重试 3-5 次。**原理**：指数退避的数学依据是——如果失败原因是服务器过载，线性重试会加重负载，而指数退避给予服务器恢复时间
3. **降级**：如果主工具失败，尝试备用工具。例如，天气 API 超时，可以尝试搜索引擎查询"北京天气"
4. **向用户解释**："天气服务暂时不可用，请稍后再试"。**关键**：解释应该包含失败原因（如"服务超时"而非模糊的"出错了"）和建议的替代方案

#### 多步调用链：复杂任务的编排

复杂任务需要多个工具的串联调用。例如："帮我规划一个从北京到上海的周末旅行，预算 2000 元"需要调用：

1. 火车票查询工具（获取车次和价格）
2. 酒店查询工具（获取住宿选项）
3. 景点查询工具（获取推荐景点）
4. 计算器工具（汇总费用并检查预算）

多步调用链的管理需要**状态机**或**有向图**来跟踪调用进度和依赖关系。这引出了一个深层问题：**LLM 适合做这种多步编排吗？**

**[合理推断]**：LLM 的 in-context learning 能力使其可以在推理过程中动态调整调用顺序——如果火车票查询返回"无票"，LLM 可以决定改为查询飞机票。这种灵活性是硬编码状态机无法提供的。但代价是：LLM 的多步推理容易**偏离目标**——在第3步时可能忘记第1步的约束（如预算限制），导致最终方案超预算。

**[开放问题]**：LLM 的多步工具调用能力和专门的规划算法（如 A* 搜索、蒙特卡洛树搜索）相比，在什么场景下各自更优？目前没有系统性的对比实验。

## 7.4 Code Generation：特殊的工具化形式——当代码成为"通用 API"

### 问题本质

Code Generation 要解决的根本问题是：**当现有工具无法满足需求时，LLM 能否临时"制造"一个工具？** 与 Function Calling 调用预定义工具不同，Code Generation 让 LLM 生成可执行代码——代码是一种"元工具"，可以表达任何可计算的任务。

### 历史约束：为什么 Code Generation 在2023年后才成熟？

- **LLM 能力约束**：2020年的 GPT-3 已经能生成简单代码，但错误率高——缺少上下文理解、类型推断不准确、无法处理复杂逻辑。2022年的 Codex（GitHub Copilot 的基础模型）首次展示了生产级代码生成能力，但仅限于短片段补全。
- **执行环境约束**：代码生成需要安全的执行环境。2022年之前，沙箱技术（如 Docker 容器隔离）已经成熟，但将 LLM 与沙箱集成的工程方案不完善——如何传递输入/输出、如何处理执行超时、如何限制资源访问等问题都需要解决。
- **安全约束**：LLM 生成的代码可能包含恶意操作（如删除文件、访问网络、窃取数据）。2023年后，OpenAI 的 Code Interpreter 和开源方案（如 Jupyter kernel 隔离）才提供了较为成熟的安全执行方案。

### 方案空间：在当时约束下，有哪些候选路径？

#### 路径A：Function Calling 为主，Code Generation 为辅

- **核心思想**：对于常见任务，使用预定义的 Function Calling；对于无法用预定义工具解决的任务，回退到 Code Generation
- **优势**：Function Calling 的可靠性高、速度快；Code Generation 作为补充处理长尾需求
- **劣势**：需要维护两套系统；何时从 Function Calling 切换到 Code Generation 的决策逻辑复杂
- **反事实推演**：如果这条路径成为主流，工具生态会高度标准化——常用操作有标准 API，罕见操作才用代码。但实践中，很多用户需求是"半结构化"的（如"分析这份数据并画图"），既不方便用预定义 API，又需要比纯代码更高级的抽象

#### 路径B：Code Generation 为主，Function Calling 为辅

- **核心思想**：默认生成代码解决问题；对于特定的高频操作（如搜索、数据库查询），使用 Function Calling 以提高效率
- **优势**：代码的通用性远超任何预定义工具集；用户不需要等待开发者创建新 API
- **劣势**：代码生成的可靠性低于 Function Calling；代码执行的安全风险更高；执行延迟更长
- **反事实推演**：如果这条路径成为主流，每个 Agent 都需要一个强大的代码执行沙箱。这会增加系统复杂度和安全攻击面。但长期来看，随着代码生成模型的质量提升和安全沙箱的标准化，这条路径可能逐渐成为主流——就像人类程序员越来越倾向于写脚本而非寻找现成工具

#### 路径C：统一为 Code Generation

- **核心思想**：将所有工具调用都视为代码生成——工具描述变为函数签名，工具调用变为函数调用代码
- **优势**：统一了接口——一切都是代码；可以利用编程语言的类型系统做参数验证
- **劣势**：对 LLM 的代码生成能力要求极高；在简单任务上，生成代码的开销（解析、编译、执行）可能超过直接输出 JSON
- **[作者推测]**：长期来看，随着 LLM 代码生成能力的提升，Function Calling 可能被 Code Generation 吸收——就像高级语言最终吸收了汇编语言的功能

#### 选择逻辑

在2023-2025年的约束下，Function Calling 和 Code Generation **共存**是主流选择。Function Calling 用于高频、标准化的操作（如搜索、天气查询、计算器），Code Generation 用于复杂的、一次性的数据处理和分析任务。这种共存不是"最优设计"，而是**工程上的折中**——在可靠性和通用性之间取得平衡。

### 7.4.1 代码作为通用工具

代码生成是工具化的一个特殊形式：与其他工具（如搜索引擎、计算器）不同，代码执行器是**通用的**——任何可以用 Python 表达的任务，都可以通过代码生成来解决。这使得代码生成成为 Agent 的"瑞士军刀"。

例如，面对"计算这组数据的均值、中位数和标准差，并绘制直方图"，Agent 可以生成 Python 代码：

```python
import numpy as np
import matplotlib.pyplot as plt

# [注意] 数据硬编码在脚本中——在真实Agent系统中应通过变量传递
data = [23, 45, 56, 78, 32, 67, 89, 12, 34, 55]
mean = np.mean(data)
median = np.median(data)
std = np.std(data)

print(f'Mean={mean:.2f}, Median={median:.2f}, Std={std:.2f}')

plt.hist(data, bins=10)
plt.title(f'Statistics Overview')
plt.savefig('histogram.png')
```

执行这段代码后，Agent 不仅获得了统计结果，还生成了可视化图表。

**[关键局限]**：这段代码有一个常见问题——数据被硬编码在脚本中。在真实 Agent 系统中，数据应该通过变量或文件传递，而非硬编码。LLM 生成代码时常犯这种错误，因为训练数据中的代码示例通常是自包含的。这是一个**分布偏差**——训练数据的分布与实际使用场景不匹配。

### 7.4.2 代码执行的安全隔离

代码执行的危险性在于：LLM 生成的代码可能包含恶意操作（如删除文件、访问网络、窃取数据）。即使 LLM 没有恶意意图，生成的代码也可能因为逻辑错误而造成损害（如无限循环消耗 CPU、内存泄漏导致 OOM）。

因此，代码执行必须在**沙箱环境**中进行：

- **容器隔离**：在 Docker 容器中执行代码，限制文件系统访问。容器提供进程级隔离，防止代码影响宿主系统
- **资源限制**：设置 CPU 时间限制（如 5 秒）、内存限制（如 256MB）。**原理**：资源限制防止代码因 bug（如无限循环）或恶意行为消耗过多资源。CPU 时间限制的5秒阈值来自经验——大多数数据分析任务在5秒内完成，超过5秒通常意味着代码有问题
- **网络隔离**：禁止代码访问外部网络。**[注意]** 某些数据分析任务需要下载数据集——可以通过预下载数据到本地目录、然后只允许读取本地文件的方式解决
- **输入消毒**：对用户输入进行消毒，防止代码注入攻击。例如，如果用户输入被直接插入到生成的代码字符串中，恶意用户可以注入任意代码

**错误驱动学习：安全沙箱的常见配置错误**

```python
# 错误示例：不安全的代码执行
import subprocess
result = subprocess.run(['python', '-c', user_provided_code], capture_output=True)
# [错误] 直接在宿主环境中执行用户代码，没有隔离
```

```python
# 正确示例：使用 Docker 容器隔离
import subprocess
result = subprocess.run([
    'docker', 'run', '--rm',  # 执行后自动删除容器
    '--memory=256m',          # 限制内存
    '--cpus=1',                # 限制 CPU
    '--network=none',          # 禁止网络访问
    '--read-only',             # 只读文件系统（写入到/tmp）
    '-v', f'{temp_dir}:/workspace:ro',  # 挂载数据目录（只读）
    'python:3.11-slim',
    'python', '-c', user_provided_code
], capture_output=True, timeout=10)  # 10秒超时
```

**错误分析**：不安全的代码执行是最常见的工程错误之一。其认知偏差根源是**乐观偏差**——"我生成的代码不会出问题"。但在 Agent 系统中，代码可能来自用户输入、LLM 生成或两者组合，任何一个环节都可能出现意外行为。

## 7.5 多模态与工具化的融合挑战——当"眼睛"和"手"需要协作

### 7.5.1 视觉-工具联合推理

当 Agent 同时拥有视觉输入和工具使用能力时，新的挑战出现：

- **视觉 grounding**：用户说"点击这个按钮"，Agent 需要理解"这个"指的是图像中的哪个按钮，然后调用点击工具。这需要**指代消解**——将语言中的指代词映射到视觉中的具体区域。当前 VLM 在这方面的能力仍然有限 [强实验证据]——VLM 能描述图像内容，但将描述与精确的空间坐标对应起来（如"按钮在 (x=234, y=567) 处"）仍然是一个开放问题
- **视觉验证**：Agent 执行操作后，需要通过视觉反馈确认操作是否成功（如"点击保存按钮后，是否出现了保存成功的提示？"）。这需要**时序视觉理解**——比较操作前后的图像变化，判断操作是否产生了预期效果。当前 VLM 主要处理静态图像，时序理解能力有限
- **跨模态一致性**：文本描述和视觉内容可能不一致（如用户说"红色杯子"，但图像中的杯子是蓝色的），Agent 需要处理这种冲突。这涉及**模态信任问题**——当文本和视觉矛盾时，Agent 应该相信哪个模态？**[合理推断]**：在安全关键场景中，视觉应该是权威（因为视觉直接反映物理世界），但在信息检索场景中，文本可能是权威（因为视觉可能是过时的截图）

### 7.5.2 学术界与工业界的断层

论文中的多模态模型通常使用精心准备的评估数据集（如 VQAv2、OK-VQA），但工业界的真实场景更加混乱：

- **图像质量参差不齐**：用户上传的图像可能模糊、低分辨率、遮挡、过度曝光。学术评估数据集中的图像通常是清晰、高分辨率、构图良好的——这种分布差异导致学术指标高估了真实场景的性能
- **用户问题含糊不清**：如"这个怎么样？"缺乏上下文。学术数据集中的问题通常明确、具体（如"图中穿红衣服的人在做什么？"）——真实用户的问题远比这模糊
- **工具 API 的响应格式不一致**：不同 API 返回的数据格式各异，需要动态解析。学术研究中的工具调用通常假设标准化的输入/输出格式

这些工程问题在论文中很少讨论，但它们是生产系统的核心挑战。**[作者推测]**：学术界和工业界之间的这个"断层"可能是当前 VLM 评估方法最大的盲区——我们需要在真实场景中评估 VLM，而非仅在标准数据集上。

### 7.5.3 批判性分析：多模态融合的内在局限

#### 内在局限性

1. **视觉理解的"浅层性"**：当前 VLM 的视觉理解可能主要是**模式匹配**而非**深层理解**。VLM 能识别"图中有猫"，但它是否理解"猫是一种会动的生物"、"猫可能跳下桌子"？**[开放问题]**——当前评估方法无法区分"识别"和"理解"
2. **空间推理的局限**：VLM 在空间关系推理（如"A在B的左边"、"C在D和E之间"）上的表现显著低于人类。这可能是投影层架构的结构性局限——线性投影丢失了空间结构信息 [强实验证据]
3. **时序理解缺失**：当前 VLM 主要处理静态图像。视频理解（时序变化、动作识别）仍然是一个活跃的研究方向，尚未成熟 [开放问题]

#### 实证盲区

- 当前 VLM 的评估数据集（如 VQAv2, GQA）主要集中在**日常场景**——猫狗、家具、户外风景。但在**专业领域**（如医学图像、工业检测、卫星图像）上的评估非常有限。VLM 在专业领域可能表现远差于预期——因为这些领域的视觉模式在预训练数据中很罕见
- **对抗性测试**：VLM 对对抗性扰动（如微小像素变化导致分类翻转）的鲁棒性研究不足。在安全关键应用（如自动驾驶、医疗诊断）中，这是一个重要的盲区

#### 替代路径的现况

- **专用视觉模型 + LLM**：对于特定视觉任务（如 OCR、人脸识别、医学图像分析），专用模型仍然优于通用 VLM。为什么？因为专用模型针对特定任务优化，而 VLM 是"万事通"——它牺牲了特定任务的极致性能换取了通用性。**[合理推断]**：在专业领域，"专用模型 + LLM"的组合可能长期优于通用 VLM
- **符号视觉推理**：一些研究者尝试将视觉信息转化为符号表示（如场景图），然后用符号推理引擎处理。这种方案在需要精确推理的任务（如空间关系推理）上可能更优，但失去了端到端训练的优势

#### 认知偏差警示

多模态研究者可能存在**通用性偏差**——倾向于认为"一个模型解决所有问题"优于"多个模型各司其职"。但这个假设并非自明——在工程实践中，模块化系统（各模块专用）通常比统一系统更容易维护、调试和升级。VLM 的"统一"优势主要体现在**跨模态推理**（如"图中的文字说了什么？"需要同时处理视觉和文本），而非单一模态任务。

## 7.6 代码：基于 LLM 的 ReAct 风格工具调用循环

下面的代码实现了一个完整的 ReAct 风格工具调用 Agent，支持多种工具（计算器、搜索模拟器、日期查询）。这个实现可以直接扩展到真实 LLM API 和真实工具。

### 错误驱动学习：先看常见错误实现

在展示正确实现之前，先看一个常见的**错误实现**及其问题：

```python
# 错误示例：不安全的工具调用循环
import json

class UnsafeAgent:
    def __init__(self, llm_api, tools):
        self.llm = llm_api
        self.tools = tools
    
    def run(self, query):
        # [错误1] 没有最大步数限制——可能无限循环
        while True:
            response = self.llm(query)
            if 'action' in response:
                tool_name = response['action']
                # [错误2] 没有验证工具是否存在
                result = self.tools[tool_name](response['args'])
                # [错误3] 没有错误处理——工具执行失败会直接崩溃
                query = query + str(result)
            else:
                return response
```

**错误分析**：
1. **无限循环风险**：没有最大步数限制，如果 LLM 反复调用工具而不给出最终答案，Agent 会无限循环
2. **缺少工具验证**：直接用 LLM 输出的工具名做字典查找，如果 LLM 幻觉出不存在的工具名，会抛出 KeyError
3. **无错误恢复**：工具执行失败时直接崩溃，没有重试或降级机制

**认知偏差根源**：这种错误实现源于**乐观偏差**——假设 LLM 的输出在任何情况下都正确、工具随时可用。在原型阶段这通常能工作（因为测试 case 简单），但在生产环境中会频繁失败。

### 正确实现

```python
import json
import re
from typing import Dict, List, Callable, Any
from dataclasses import dataclass

@dataclass
class Tool:
    """工具定义"""
    name: str
    description: str
    parameters: Dict[str, Any]
    function: Callable

class SimpleAgent:
    """
    ReAct 风格工具调用 Agent。
    模拟 LLM 的推理-行动循环，使用规则系统替代真实 LLM API。
    
    [注意] 这是一个教学实现。生产系统中应使用真实 LLM API + 完善的错误处理。
    [关键局限] 规则系统只能处理预定义模式，无法泛化到新场景——
    这恰恰模拟了 LLM 在工具调用上的局限：模式匹配 ≠ 真正的推理。
    """
    def __init__(self, tools: List[Tool], max_steps: int = 10):
        self.tools = {tool.name: tool for tool in tools}
        self.max_steps = max_steps  # [关键] 防止无限循环
        self.memory = []  # 对话历史
    
    def simulate_llm_reasoning(self, query: str, available_tools: List[str]) -> Dict[str, Any]:
        """
        模拟 LLM 的推理过程。在真实系统中，这里调用 LLM API。
        """
        query_lower = query.lower()
        
        # 规则：检测需要计算的问题
        if any(op in query for op in ['+', '-', '*', '/', 'calculate', 'sum', 'compute']):
            expr_match = re.search(r'[\d\s+\-*/().]+', query)
            if expr_match:
                expr = expr_match.group().strip()
                return {
                    'thought': f'I need to calculate "{expr}". I will use the calculator tool.',
                    'action': 'calculator',
                    'arguments': {'expression': expr}
                }
        
        # 规则：检测日期查询
        if any(word in query_lower for word in ['date', 'today', 'time', 'day']):
            return {
                'thought': 'The user is asking about the current date. I will use the date tool.',
                'action': 'date_query',
                'arguments': {}
            }
        
        # 规则：检测搜索需求
        if any(word in query_lower for word in ['search', 'find', 'look up', 'information about']):
            keywords = query_lower.replace('search', '').replace('find', '').replace('for', '').strip()
            return {
                'thought': f'I need to search for "{keywords}". I will use the search tool.',
                'action': 'search',
                'arguments': {'query': keywords}
            }
        
        # 默认：直接回答
        return {
            'thought': 'This is a general question. I can answer directly without tools.',
            'action': 'direct_answer',
            'arguments': {'query': query}
        }
    
    def execute_tool(self, tool_name: str, arguments: Dict[str, Any]) -> str:
        """执行工具调用，包含完善的错误处理"""
        # [关键] 验证工具是否存在
        if tool_name not in self.tools:
            return f"Error: Tool '{tool_name}' not found. Available tools: {list(self.tools.keys())}"
        
        tool = self.tools[tool_name]
        try:
            # [关键] 验证必填参数
            for param_name, param_info in tool.parameters.items():
                if param_name not in arguments:
                    return f"Error: Missing required parameter '{param_name}'"
            
            result = tool.function(**arguments)
            return str(result)
        except Exception as e:
            # [关键] 捕获异常并返回结构化错误信息
            return f"Error executing tool '{tool_name}': {str(e)}"
    
    def process(self, query: str) -> Dict[str, Any]:
        """
        处理用户查询，执行完整的 ReAct 循环。
        包含步数限制和错误恢复。
        """
        # [关键] 步数限制防止无限循环
        for step in range(self.max_steps):
            # 1. 推理
            reasoning = self.simulate_llm_reasoning(query, list(self.tools.keys()))
            thought = reasoning['thought']
            action = reasoning['action']
            arguments = reasoning['arguments']
            
            # 2. 行动
            if action == 'direct_answer':
                observation = "Answered directly."
                # 直接回答后退出循环
                final_answer = f"Based on my knowledge: {query}"
                break
            else:
                observation = self.execute_tool(action, arguments)
                
                # [关键] 检查是否需要继续推理
                if observation.startswith('Error'):
                    # 工具调用失败——可以选择重试或直接返回错误
                    final_answer = f"I encountered an issue: {observation}"
                    break
            
            # 3. 生成最终回答（模拟）
            if action == 'calculator':
                final_answer = f"The result is {observation}."
                break
            elif action == 'date_query':
                final_answer = f"Today is {observation}."
                break
            elif action == 'search':
                final_answer = f"I found the following information: {observation}"
                break
        else:
            # [关键] 超过最大步数仍未完成
            final_answer = "I was unable to complete this task within the allowed steps."
        
        # 记录到记忆
        self.memory.append({
            'query': query,
            'thought': thought,
            'action': action,
            'arguments': arguments,
            'observation': observation,
            'answer': final_answer
        })
        
        return {
            'thought': thought,
            'action': action,
            'arguments': arguments,
            'observation': observation,
            'answer': final_answer,
            'step': step + 1,
            'memory_size': len(self.memory)
        }

# ============ 工具实现 ============
def calculator(expression: str) -> float:
    """安全计算器：只支持基本算术
    [注意] eval() 在生产环境中不安全。应使用 ast.literal_eval 或专门的数学解析库。
    """
    allowed = set('0123456789+-*/(). ')
    if not all(c in allowed for c in expression):
        raise ValueError("Invalid characters in expression")
    return eval(expression)

def search_simulator(query: str) -> str:
    """模拟搜索引擎"""
    mock_results = {
        'python': 'Python is a high-level programming language created by Guido van Rossum.',
        'transformer': 'Transformer is a neural network architecture introduced by Vaswani et al. in 2017.',
        'llm': 'LLM stands for Large Language Model, a type of AI model trained on vast text data.'
    }
    for key, value in mock_results.items():
        if key in query.lower():
            return value
    return f"Search results for '{query}': No relevant information found."

def date_query() -> str:
    """返回当前日期"""
    from datetime import datetime
    return datetime.now().strftime("%Y-%m-%d %A")

# ============ 演示 ============
if __name__ == "__main__":
    tools = [
        Tool('calculator', 'Perform arithmetic calculations', 
             {'expression': 'string'}, calculator),
        Tool('search', 'Search for information', 
             {'query': 'string'}, search_simulator),
        Tool('date_query', 'Get current date', 
             {}, date_query)
    ]
    
    agent = SimpleAgent(tools, max_steps=10)
    
    queries = [
        "What is 37 + 15 * 2?",
        "What is the date today?",
        "Search for information about Python",
        "Tell me a joke"
    ]
    
    for query in queries:
        print(f"\n{'='*50}")
        print(f"User: {query}")
        result = agent.process(query)
        print(f"Thought: {result['thought']}")
        print(f"Action: {result['action']}")
        print(f"Observation: {result['observation']}")
        print(f"Answer: {result['answer']}")
    
    print(f"\n{'='*50}")
    print(f"Memory recorded {len(agent.memory)} interactions.")
```

### 代码解析

这个 ReAct 工具调用实现展示了几个关键概念：

1. **工具注册**：通过 `Tool` 数据类定义工具的名称、描述、参数模式和执行函数。工具描述在真实系统中会被传递给 LLM，作为 Function Calling 的 schema。
2. **推理模拟**：`simulate_llm_reasoning` 用规则系统模拟 LLM 的决策过程。在真实系统中，这里应调用 LLM API，传入工具描述和当前对话，让 LLM 生成 Thought + Action + Arguments。
3. **安全执行**：`calculator` 工具通过字符白名单和 `eval` 实现安全计算。在真实系统中，应使用更安全的沙箱（如 `ast.literal_eval` 或 Docker 容器）。**[注意]** `eval` 即使有字符白名单也并非完全安全——某些边界 case（如超长表达式导致 DoS）仍可能被利用。
4. **错误恢复**：`execute_tool` 捕获异常并返回错误信息，Agent 可以在下一轮推理中处理错误（如重试或向用户解释）。与错误实现的关键区别在于：错误不会导致 Agent 崩溃。
5. **步数限制**：`max_steps` 防止无限循环——这是一个看似简单但至关重要的工程实践。在生产系统中，建议设置 5-10 步的上限。
6. **记忆管理**：每次交互记录到 `memory` 中，支持多轮对话的上下文保持。

## 7.7 工程实践框：多模态与工具化的部署要点

### 常见陷阱

**1. 视觉输入的分辨率陷阱**
- **现象**：许多 VLM 使用 224×224 或 336×336 的固定输入分辨率。如果用户上传 4K 图像，直接 resize 会导致文字模糊、小物体消失
- **解决方案**：使用图像分块（将大图切分为多个子图分别编码）或高分辨率适配器（如 LLaVA-1.5 的 AnyRes）
- **认知偏差根源**：**锚定偏差**——开发者在测试时使用的是标准数据集图像（通常 224×224），因此以这个分辨率为锚点设计系统。但真实用户的图像分辨率分布远比标准数据集分散

**2. 工具描述的上下文占用**
- **现象**：当可用工具很多时（如 100+ 个 API），工具描述会占用大量上下文空间，导致用户输入空间被压缩。每个工具的 JSON Schema 描述约占用 100-300 个 token，100 个工具就是 10,000-30,000 个 token
- **解决方案**：使用工具检索（根据用户查询动态选择最相关的工具，只传入这些工具的描述）或工具分层（将工具分类，先选择类别再选择具体工具）
- **认知偏差根源**：**可用性偏差**——因为所有工具都可用，所以倾向于全部传入。但在实践中，大多数查询只需要 2-5 个工具

**3. Function Calling 的 JSON 解析失败**
- **现象**：LLM 生成的 JSON 可能包含语法错误（如缺少引号、多余的逗号、在 JSON 中混入自然语言注释）
- **解决方案**：使用 JSON 修复库（如 `json_repair`）或结构化输出（如 OpenAI 的 JSON mode、Outlines 库）。**原理**：JSON mode 通过在解码时约束 token 选择，确保输出是合法 JSON——这是在概率分布层面解决问题，而非在后处理层面
- **认知偏差根源**：**可靠性幻觉**——因为 Function Calling 在常见 case 上的成功率 > 95%，开发者倾向于假设它在任何情况下都可靠。但 5% 的失败率在生产环境中意味着每 20 次调用有 1 次出错

**4. 多模态的 token 成本**
- **现象**：视觉 patch 的数量通常很大（如 336×336 图像用 14×14 patch，有 576 个视觉 token）。如果按 token 计费，视觉输入的成本可能是文本的 10-100 倍
- **解决方案**：压缩视觉 token（如使用视觉摘要模型提取关键 patch）或降低图像分辨率（对于简单任务，224×224 通常足够）
- **认知偏差根源**：**隐性成本忽视**——开发者在测试时使用少量图像，成本不明显。但在生产系统中，每天处理数万张图像时，视觉 token 成本会成为主要开支

### 硬件需求估算

| 配置 | 硬件要求 | 适用场景 | 成本（2025年中参考） |
|------|---------|---------|-------------------|
| VLM 推理（LLaVA-7B，336×336） | RTX 4090 24GB | 原型开发、小规模测试 | ~$1,600（GPU） |
| VLM 推理（LLaVA-13B，336×336） | RTX 4090 24GB（4-bit量化）或 A100 40GB | 中等规模部署 | $1,600-$10,000 |
| VLM 微调（LLaVA-7B，投影层） | A100 40GB ×1 | 定制化VLM | ~$10,000（GPU） |
| VLM 全参数微调（LLaVA-7B） | A100 80GB ×2-4 | 大规模定制 | $20,000-$40,000 |
| 工具调用 Agent（GPT-4 API） | 无GPU需求（API调用） | 通用Agent | $0.01-0.50/次任务 |
| Code Generation 执行 | CPU + 256MB内存（沙箱） | 代码执行 | ~$0（本地资源） |

**[时间敏感信息]** 上述成本基于2025年6月市场价格。GPU 短缺和云服务商定价策略可能导致实际成本波动 ±30%。API 成本随提供商定价策略变化。

**经验法则**：
- 简单工具调用（1-2 步）：API 成本约 $0.01-0.05
- 复杂任务（5-10 步）：API 成本约 $0.10-0.50
- 需要数据分析（如 Pandas 处理 1GB CSV）：沙箱需 2-4 GB 内存和 10-30 秒 CPU 时间

### 超参选择建议

| 超参 | 建议值 | 原理说明 |
|------|--------|--------|
| VLM 图像分辨率（简单分类/描述） | 224×224 | 低分辨率足够捕捉主要语义信息；计算成本最低 |
| VLM 图像分辨率（文档理解/OCR） | 336×336 或更高 | 文字识别需要更高分辨率；224×224 会导致小字模糊 |
| VLM 图像分辨率（医学图像/精细推理） | 448×448 或分块处理 | 需要精细视觉特征；分块处理可避免单次计算量过大 |
| 工具选择阶段温度参数 | T=0.1-0.3 | 低温度确保参数填充的准确性——工具调用需要确定性，不需要创造性 |
| 工具结果总结阶段温度参数 | T=0.7-0.9 | 高温度增加回答的自然性——总结工具结果时需要一定的语言多样性 |
| 上下文窗口管理策略 | 摘要 + 滑动窗口 | 当对话历史 + 工具描述 + 工具结果超过上下文限制时，对历史对话摘要并保留最近 N 轮 |
| 重试策略 | 指数退避，3-5次 | 间隔 1s, 2s, 4s, 8s——给予服务恢复时间，同时避免无限等待 |
| 代码执行超时 | 5-10秒 | 大多数数据分析任务在5秒内完成；超过10秒通常意味着代码有bug或需要优化 |
| Agent 最大步数 | 5-10步 | 防止无限循环；大多数任务在5步内完成，超过10步通常意味着任务超出Agent能力 |

> **关键提醒**：多模态和工具化让 LLM 从"对话者"变成了"行动者"。但"能行动"不等于"能可靠行动"——在生产环境中，可靠性来自工程实践（错误恢复、降级策略、步数限制），而非模型能力本身。当你设计一个多模态 Agent 时，请习惯性地追问："如果这个模态/工具失败了，系统会怎样？"——如果答案是"崩溃"，你的设计还需要更多的工作。
